两数之和

题号：1
描述：
给定一个整数数组 nums 和一个整数目标值 target，请你在该数组中找出 和为目标值 的那 两个 整数，并返回它们的数组下标。你可以假设每种输入只会对应一个答案。但是，数组中同一个元素不能使用两遍。你可以按任意顺序返回答案。
示例1：
输入：nums = [2,7,11,15], target = 9
输出：[0,1]
解释：因为 nums[0] + nums[1] == 9 ，返回 [0, 1] 。

题解一：

​ 首先想到的肯定是暴力破解法，通过题目的描述，我们仅需要将数组中的数字两两组合（不需要重复组合），并且判断和是否为target即可。

题解二：

​ 题目中，我们每得到一个数字，就需要与数组中数字进行判断和是否为target。我们可以借助哈希表，实现快速取数并进行比对。

  1. 遍历nums数组
  2. 在哈希表中查找，是否有target-nums[i]的值，有则返回i和哈希表target-nums[i]的值，没有则存入nums[i]和下标i。

整数反转

题号：7

描述：给你一个 32 位的有符号整数 x ，返回 x 中每位上的数字反转后的结果。

如果反转后整数超过 32 位的有符号整数的范围 [−2^31^, 2^31^ − 1] ，就返回 0。

假设环境不允许存储 64 位整数（有符号或无符号）。

示例1：

输入：x = 123
输出：321

题解一：

​ 提取出每一位的数字，并重新组合。

  1. 将x%10得到个位值，并加上result*10
  2. 将x / 10舍去个位数字，并将十位数字变为各位数字，同步骤1操作。
  3. 当x为0时，循环结束
  4. 判断结果是否溢出，在循环中，result只要大于int类型最大值/10,或者小于int类型最小值/10都会溢出。

题解二：

​ 与题解一思路相同，这里提供另外一种判断溢出的方法。int类型占4字节会有溢出，将结果给一个long类型的变量（占8字节），最终结果将long类型强制转换为int类型并且与原long类型比较大小，若相同则没有溢出，若不相同则溢出。

5.安全性测试

1.输入的数据没有进行有效的控制和验证

​ a.登录必须输入邮箱，如果格式不对，不会进行报错。

​ b.密码是以小圆点表示，但是没有查看密码按钮。

图二十九 账号密码

​ c.不允许在没有输入账号密码的情况下登录。

图三十 账号密码提示

2.用户名和密码

​ a.接口程序连接登录时，没有验证码或者图片验证。容易被攻击者反复试验其密码。

​ b.没有密码错误次数机制。在短时间内应当记录密码错误次数，进行账号保护，或者延时再登录等保护机制。盗号风险较高。

​ c.忘记密码，没有找回密码的方式。注册为邮箱注册，应该至少提供一种邮箱找回密码的方式。这样会导致账号废弃率很高，导致数据库无用账号太多（用户忘记密码后，只能重新创建一个账号继续使用服务），加大数据库的响应速度和查询时间。

​ d.注册账号时，邮箱验证流程不对，该网站是（先注册成功），在登录后进行邮箱验证。这样会导致攻击者，反复注册，但不验证，让数据库的数据量撑爆，使数据库瘫痪。正确流程应该是先邮箱认证通过后，再创建账号。

​ e.修改密码时，同样没有密码规范，甚至可以设置一位数密码。让整个系统的密码质量低，盗号风险非常高。

3.直接输入需要权限的网页地址可以访问

​ a.直接输入登录后的url，不能进入网页，会跳转到登录页面。

​ b.退出账号后，点击浏览器的后退，还是能进入到退出之前的页面。加大他人盗取账号信息的风险。

4.操作时间的时效性

​ 该网站操作没有时效性，不会自动退出登录或者验证等操作。当你离开电脑或者忘记退出时，有他人盗取信息的风险。理想情况下应当设置失效时间，当未操作网站的时间达到失效时间，再需要进行操作时，需要验证。

五、总结

​ 经过这次测试，天气API网站基本符合要求，但还存在很多界面上、安全上的问题：

  • 在浏览器的缩放过程中，会挤压一些元素，导致页面变形，或者挤压到第二排。各个页面都这这个问题。这样会导致手机访问网站时会有很差的布局和体验；

  • 在导航栏的操作部分用户体验一般，在网页底部不能快速的操作导航栏。

  • 密码没有字符长度等具体限制，密码强度会较弱，加大盗号风险。

  • 注册账号时，邮箱验证流程不对，该网站是（先注册成功），再登录后进行邮箱验证。这样会导致攻击者，反复注册，但不验证，让数据库的数据量爆增，使数据库瘫痪。正确流程应该是先邮箱认证通过后，再创建账号。
  • 登录时，在密码输入错误的情况下，提示信息“密码错误”，会很不安全，让攻击者知道该账号是存在的，只是密码不对，大大加大了盗号风险。更好的提示应为“账号或者密码错误”。
  • 在整个注册和登录的页面，并未有“忘记密码”按钮，当用户忘记密码时，此账号就不能进行操作，成为废账号，而用户只能创建新账号，这会大大加大数据库的数据量，使系统查询会越来越慢。
  • 密码修改完成后并没有重新登录，而是在界面继续操作。
  • 支付过程，很容易就被抓包获取订单，不太安全。建议参考一下其他网站收费流程。
  • 性能测试中，在用户数量增加时，事务的响应时间也在同步增长，在40-50秒时剧增。由此看来对服务器造成了高度负载，运行不太稳定。
  • 密码输入时，没有查看密码按钮。
  • 接口程序连接登录时，没有验证码或者图片验证。容易被攻击者反复试验其密码。
  • 没有密码错误次数机制。在短时间内应当记录密码错误次数，进行账号保护，或者延时再登录等保护机制。盗号风险较高。
  • 退出账号后，点击浏览器的后退，还是能进入到退出之前的页面。加大他人盗取账号信息的风险。
  • 该网站操作没有时效性，不会自动退出登录或者验证等操作。当你离开电脑或者忘记退出时，有他人盗取信息的风险。理想情况下应当设置失效时间，当未操作网站的时间达到失效时间，再需要进行操作时，需要验证。
  • 移动端兼容性很差。原因是页面布局时并未考虑手机端的浏览。可以优化页面布局，即可以很好的兼容手机浏览器。

困难：

​ 其实不管是界面模块、图像处理的功能模块还是视频处理模块，都没有那么顺畅的完成，大大小小很多可能印象不是很深了，所以列举一些印象深一点的困难。

Sobel边缘检测：这一类还包括高斯滤波、直方图均值化等需要浮点数类型的处理。在方法实现中稍微没有控制好或者考虑到数据类型转换问题，就会精度下降或者图片全白全黑等问题。

中值滤波：我实现的滤波基本都是四层循环，外面两层遍历像素点，里面两层遍历模板，而中值滤波需要对模板点排序，所以我又实现了一个冒泡排序，导致一共六层循环，复杂度很高。在3x3等小一点的模板示例图片速度还行，但是当模板变大一点，循环次数是爆炸性的增长，所以处理时间也会非常长。

视频处理：在对播放帧的处理是，将视频转化为一帧帧图片，通过我显示图像的Label显示，同时存入我的QImage数组，我想暂停后对这一帧进行图片处理后再存入QImage数组，但是怎么也实现不了，因为处理完成之前，图片已经存入了，你修改后在label显示的图片只会被后面帧的图片冲掉。最后我的解决方案是定义int counters[100]和QImage editeImageVideo[100]两个数组存放修改的帧数和这一帧修改后的图像，当我点击预览的时候，将这些修改的帧插入存视频帧的QImage数组后，再播放，实现了播放修改后的视频；

视频修改后预览：我是遍历QImage数组里面的图像信息，显示到Label，这样界面的情况是会直接显示最后一张图片，原因是因为遍历数组太快了。所以我在遍历中用到计时器，通过获取视频总时长除以总帧数得到每一帧的间隔，在遍历时加上这一间隔的时间，最后能达到还原播放的效果；
