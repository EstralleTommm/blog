    copy_to_user(void __user to, const void from, unsigned long n)；

    • to :目标地址（用户空间指针）

思考：

​ 我上面的方式是直接在内核中定义一个char[20]的缓存，而call系统调用中的参数是一个void类型指针，所以我想把内核缓存也定义为一个void类型指针data，这个时候就有一个内存分配的问题。我就通过调用void *kmalloc(size_t size, gfp_t flags);将用户传入的data_len为data分配内存，在服务结束时通过void kfree(const void *objp);将分配的内存释放；

​ 另一个值得注意的地方是copy_from_user和copy_to_user的第三个参数是拷贝数据的字节数，所以用户程序通过call系统调用时传入的参数data_len应该为传入数据的字节数；

​ 后面交流中发现，老师更偏向于静态分配一个buffer。原因是我虽然解决了buffer不够的情况，但是对于内核来说，动态分配很容易造成大量的内存碎片；另一方面，一般通信时数据包都会有大小限制；果然都是我没考虑到的地方。晚上睡觉的时候又想了想，类似于线程池产生原因，如果用户很多的情况下，每次进行动态分配再回收这样的模式，可能会占用CPU大量的处理时间，导致服务无法完成的情况。

​ 上面的思考可能很多都是我的主观见解，不一定正确，欢迎讨论！
